# 專案知識庫（核心規範）

> 本檔為本專案 AI 協作規範的**唯一真實來源**。
> `CLAUDE.md`（Claude Code）與 `AGENTS.md`（Codex）皆指向本檔，請勿在那兩份重複撰寫內容，一律只改本檔。

## 專案用途

- 本專案是「實價登錄核定訴訟標的價額工具」。
- 用途：協助以官方實價登錄資料，產出查詢截圖與佐證資料。
- 截圖為主要依據，CSV/JSON 僅作輔助整理。

## 協作規範

- 全程使用繁體中文回覆。
- 法律內容、訴訟標的價額判斷、官方資料來源不得臆測；涉及法律或官方資料時，必須查證來源。
- 優先採用官方資料來源（如內政部不動產交易實價查詢服務網、司法院／全國法規資料庫）。
- 不得把終端機顯示的中文亂碼，直接判定為檔案損壞；本專案中文檔名與文件內容可能是正常 UTF-8。
- 編輯含中文的檔案時，請確保存成 **UTF-8（無 BOM）**，勿轉成 Big5 或加上 BOM。

## 專案慣例

- 案件設定檔慣例為 `案件設定*.json`。
- 輸出檔案慣例放在 `output/`。
- 驗證工具是否可執行時，使用 `npm run evidence`。
- `排除房地車` 預設為 `false`（納入房地車，理由見下方標準作法）。

## MCP 工作流

本專案提供 MCP server（`mcp/lvr-mcp-server.mjs`），讓 AI 助理可直接觸發官方查詢截圖流程與 Open Data 預覽。

### 工具清單

#### v1 — 官網截圖流程

| 工具 | 用途 |
| --- | --- |
| `run_official_capture(configFile)` | 依案件設定檔，背景啟動官網查詢截圖，立刻回傳 runId |
| `list_evidence_runs()` | 列出歷次輸出資料夾，標示「完成／執行中／查詢未成功」 |
| `read_evidence_summary(runId)` | 讀取某次查詢的結果摘要（操作紀錄、SHA-256、最接近比較標的） |

#### v2 — Open Data 預覽（不打官網）

| 工具 | 用途 |
| --- | --- |
| `refresh_lvr_open_data()` | 從內政部 Open Data 下載最新整批買賣 CSV ZIP 至 `cache/open-data/`，記錄 SHA-256 |
| `preview_comparables(...)` | 從本機快取 CSV 篩選比較標的，回傳筆數、前幾筆、門牌彙總（不打官網） |

### 典型流程（v2 完整流程）

1. 呼叫 `refresh_lvr_open_data` 下載或更新 Open Data 快取（僅需偶爾更新）。
2. 呼叫 `preview_comparables` 帶入縣市、行政區、門牌關鍵字等條件，預覽篩選。
3. AI 與使用者討論條件、調整篩選。
4. 條件定稿後，AI 或人工備好 `案件設定_某案.json`。
5. 呼叫 `run_official_capture` 正式進官網截圖**（只打一次）**。
6. 用 `read_evidence_summary` 查結果。

> ⚠️ 預覽結果與官網查詢不會逐筆完全相同（資料更新時點不同），預覽僅供「篩選與討論」，最終佐證以官網截圖為準。

### Open Data 預覽限制（重要）

- `preview_comparables` 是 Open Data 預覽，**不是正式證據**。
- Open Data 快取可能只涵蓋特定期別（通常為最新一季），**不一定等於完整歷史資料**。
- `preview_comparables` 回傳的 `dataCoverage` 欄位會標示快取涵蓋的交易日期範圍；查詢期間超出快取涵蓋期間時，查無資料**不能直接解讀為「官方無交易」**。
- 最終佐證仍以 `run_official_capture` 產出的**官網截圖與操作紀錄**為準。

### MCP 安全設計（勿繞過）

- **路徑安全**：`run_official_capture(configFile)` 僅接受專案根目錄內符合 `案件設定*.json` 的檔名，不允許 `..` 路徑跳脫或絕對路徑。`read_evidence_summary(runId)` 僅接受單一資料夾名稱，不允許斜線、反斜線或 `..`，解析後必須位於 `output/evidence/` 內。
- **強制 headless、不保留視窗**：MCP 模式以 `LVR_HEADLESS_FORCE=1` 覆蓋設定檔的 `完成後保留視窗`／`顯示瀏覽器`，避免 tool call 等人工關窗而卡死。
- **不污染 stdout**：MCP server 本身不 `console.log`（會破壞 JSON-RPC）；子程序輸出全導向各 run 資料夾的 `_執行log.txt`。
- **fire-and-return**：`run_official_capture` 注入 `LVR_RUN_ID` 後在背景啟動子程序，立刻回傳 runId，不同步等查詢跑完。「狀態」就是 `output/evidence/<runId>/` 資料夾本身，不另設 job queue。
- **官網只打一次**：官網對短時間重複查詢會節流，MCP 流程設計為「條件定稿後，只對官網正式跑一次」。
- **preview 不打官網**：`preview_comparables` 僅讀本地快取的 Open Data CSV，不連官方查詢網站。

### 連接方式

- Claude Code：專案根目錄已附 `.mcp.json`（內含本機絕對路徑，不入版控）。
- 其他 MCP client：以 STDIO 啟動 `node mcp/lvr-mcp-server.mjs`（或 `npm run mcp`）。
- `.mcp.json` 內為本機絕對路徑，換機器需自行調整。

## 標的價額分析標準作法（核心方法）

處理「核定訴訟標的價額」找比較標的時，依下列流程：

1. **查詢條件**：以系爭房屋的**門牌／巷弄**為關鍵字（社區名常查無，勿用社區名）。交易期間以**起訴時點所在的一年內**為主（價額看的是起訴時，跨好幾年的舊資料不實用）；若要確認系爭房屋本身有無成交，可先放寬期間。
2. **一律納入房地車（不要排除）**：實測官方資料中，房地車成交的「總價／總面積／單價」**已是房地部分**，車位價與車位面積另計於「車位總價」欄；因此**含車位成交的單價與純房地一致，可直接比較，不需再扣車位**。排除房地車會白白丟掉大半市場（大直一帶約七成成交含車位），而且常把**系爭房屋本身**的成交也濾掉。
3. **比較標的優先序**：同棟（同門牌）＞ 同巷（本巷）＞ 鄰近他巷／他社區。**不同社區單價水準差異大，勿混比**（例：161巷約 90~118 萬/坪 vs 鄰巷 162巷美堤花園約 67 萬/坪）。
4. **排除或保守看待**：一樓、商業用、超大坪數／複層大戶（型態不同）；有增建（陽台外推、頂樓加蓋、其他增建）會灌高單價，需標註並保守。
5. **以起訴年月為時間錨**，挑時間最接近者；呈現**單價區間**供參。
6. **先看系爭房屋本身有無成交**（最直接證據）；沒有才用上述比較標的。
7. **分際**：面積未知時不換算總價額；訴訟標的價額之核定**屬法院職權**，只呈現官方資料與比較分析供參，**不下最終定論數字**。

## 官方資料來源備註（已查證，勿重複試錯）

- 官方「**匯出列表／清單列印** Excel」綁在 `list.jsp` 結果頁流程，且該流程有 `if(webdriver) return` 反自動化偵測、縣市選單冷啟動不載入，**無法穩定自動下載**。需要官方 Excel 時，於「完成後保留視窗」的官方頁面**人工點列印**即可。
- 備援官方資料檔：內政部 Open Data（`https://plvr.land.moi.gov.tw/Download?type=zip&fileName=lvr_landcsv.zip`，CSV/UTF-8、無反爬蟲），但為**整縣市整期**的整批資料、需自行依門牌篩選，且歷史為每季一包。
- 給 AI 分析用的文字檔，直接用工具自動產出的 `官方查詢結果_全部.csv`／`.json` 即可，不必另抓。

## 修改與驗證

- 修改查詢、自動化、截圖、輸出格式相關程式後，應盡可能實際執行 `npm run evidence` 驗證。
- 修改 MCP server 後，至少用 JSON-RPC handshake 確認五個 tool 正常註冊且 stdout 無污染。
- 不要任意改動既有輸出檔名、欄位、資料夾結構，除非使用者明確要求。
